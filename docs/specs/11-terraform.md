## Overview

Verify and harden the full Terraform stack so the entire Sentinel platform is
reproducible from a clean checkout. The `infra/` files are already logically
separated; this feature audits them for correctness, removes any hardcoded
identifiers, confirms no sensitive state is tracked in git, and validates that
`terraform plan` shows zero drift against live infrastructure.

## Requirements

1. All `infra/*.tf` files are logically organized — one concern per file:
   `provider.tf`, `ecr.tf`, `ec2.tf`, `sns.tf`, `observability.tf`,
   `lambda.tf`.
2. `*.tfstate` and `*.tfstate.backup` are gitignored and NOT tracked in git.
3. `.terraform.lock.hcl` IS committed (pins provider versions for
   reproducibility).
4. `lambda/remediation.zip` is NOT committed (it is generated by the
   `archive_file` data source at plan time and is covered by `*.zip` in
   `.gitignore`).
5. Every `variable` block has a `description` attribute. The `aws_region`
   variable in `provider.tf` is currently missing one.
6. No hardcoded AWS account IDs in any `.tf` file. The offender is
   `lambda.tf` lines 69–70, where the CloudWatch log group ARNs embed the
   literal account ID `262439760394`. Replace with
   `data.aws_caller_identity.current.account_id`.
7. No hardcoded region strings inside resource ARNs. The offender is
   `lambda.tf` line 44, where the SSM document ARN embeds `ap-south-1`
   literally. Replace with `var.aws_region` (or add
   `data.aws_region.current` and use `.name`).
8. `terraform validate` passes with no errors.
9. `terraform plan` shows zero changes after `terraform apply` (no drift).

## Out of scope

- Remote state backend (S3 + DynamoDB). This is the correct production
  upgrade — a single S3 bucket for state plus a DynamoDB table for lock
  management — but is out of scope for a portfolio project. Note: this
  DynamoDB lock table is completely distinct from the deduplication DynamoDB
  that was deliberately cut from the alarm design (Feature 9). They solve
  different problems; don't conflate them in interviews.
- Terraform modules or workspaces.
- Any new AWS resources.
- Migrating SNS email endpoints or other non-secret configuration values to
  variables (acceptable to leave inline for a portfolio project).

## Acceptance criteria

- `git ls-files 'infra/*.tfstate'` returns empty (state not tracked).
- `git ls-files infra/.terraform.lock.hcl` returns the file (lock committed).
- `git ls-files 'lambda/*.zip'` returns empty (build artifact not tracked).
- `grep -rE '[0-9]{12}' infra/*.tf` returns empty (no hardcoded account IDs).
- Every `variable` block in `infra/*.tf` contains a `description` line.
- `terraform validate` exits 0 with "Success! The configuration is valid."
- `terraform plan` exits 0 and reports "No changes. Your infrastructure is
  up-to-date."

## Notes

- **Local state is intentional for a portfolio project.** The production
  upgrade is an S3 backend with a DynamoDB lock table; frame this as a
  deliberate trade-off, not an oversight, if asked in interviews.
- **No IAM changes needed.** This feature is purely verification and
  cleanup — no new AWS resources, no new permissions.
- The `aws_caller_identity` data source requires no IAM permissions; it
  calls `sts:GetCallerIdentity`, which is allowed for any authenticated
  principal by default.
- After replacing hardcoded values, re-run `terraform plan` to confirm the
  substitution produces no resource replacement (these are only used inside
  inline policy strings, so no resource IDs change).
